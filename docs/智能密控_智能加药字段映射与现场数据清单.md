# 智能密控、智能加药字段映射与现场数据清单

## 1. 文档目的

这份文档用于现场对接时统一口径，明确：

- 平台接口传什么
- Python 模型实际要什么
- 每个字段对应什么现场数据
- 还缺哪些现场确认项

结论先说：

- 前端页面里的演示模板不能直接作为现场采集口径。
- 现场应以 Python 模型真实入参结构为准。
- 智能密控、智能加药都存在“平台示例字段”和“算法真实字段”不一致的问题，现场要数时必须按本文件执行。

## 3. 平台统一接口格式

### 3.1 智能密控

平台侧请求：

```json
{
  "unit": "3207",
  "dataLong": [],
  "dataShort": [],
  "params": {}
}
```

后端转发给 Python：

```json
{
  "data_long": [],
  "data_short": [],
  "para": {}
}
```

### 3.2 智能加药

平台侧请求：

```json
{
  "unit": "601",
  "dataLong": [],
  "dataShort": [],
  "params": {}
}
```

后端转发给 Python：

```json
{
  "data_long": [],
  "data_short": [],
  "para": {}
}
```

## 4. 智能密控字段映射

### 4.1 适用单元

- `3207`
- `3208`
- `316`

### 4.2 `data_long` 映射

| 下标 | Python字段 | 含义 | 数据类型 | 现场建议来源 |
| --- | --- | --- | --- | --- |
| `data_long[0]` | `rho` | 密度历史序列 | 数值数组 | 密度计实时/历史 |
| `data_long[1]` | `yw` | 液位历史序列 | 数值数组 | 液位计实时/历史 |
| `data_long[2]` | `diverter` | 分流阀开度历史序列 | 数值数组 | 分流阀反馈 |
| `data_long[3]` | `PDC` | 皮带秤历史序列 | 数值数组 | `3203` 末煤皮带秤 |
| `data_long[4]` | `QT` | 启停信号历史序列 | 数值数组 | 单元运行/停机信号 |

说明：

- 算法实际只读取了 `data_long[0..4]`。
- 页面演示模板中 `dataLong` 放了 8 个标量示例值，这和算法真实结构不一致。
- 现场应提供“按时间顺序的数组”，不是单个瞬时值。

### 4.3 `data_short` 映射

#### `3207`、`3208`

| 下标 | Python字段 | 含义 | 数据类型 | 条件 |
| --- | --- | --- | --- | --- |
| `data_short[0]` | `water` | 默认补水反馈 | 数值数组或短窗数组 | `water_switch=0` 时使用 |
| `data_short[1]` | `JJB` | 加介泵反馈 | 数值数组或短窗数组 | 必填 |
| `data_short[2]` | `yw_XJ` | 稀介桶液位反馈 | 数值数组或短窗数组 | 必填 |
| `data_short[3]` | `water` | 备用补水反馈 | 数值数组或短窗数组 | `water_switch=1` 时使用 |

#### `316`

| 下标 | Python字段 | 含义 | 数据类型 | 条件 |
| --- | --- | --- | --- | --- |
| `data_short[0]` | `water` | 默认补水反馈 | 数值数组或短窗数组 | `water_switch=0` 时使用 |
| `data_short[1]` | `JJB` | 加介泵反馈 | 数值数组或短窗数组 | 必填 |
| `data_short[2]` | `water` | 备用补水反馈 | 数值数组或短窗数组 | `water_switch=1` 时使用 |

说明：

- `316` 和 `3207/3208` 的 `data_short` 结构不完全相同。
- `316` 没有读取 `yw_XJ`。
- 现场必须分别确认三套单元的短周期点位，不要按同一模板硬套。

### 4.4 `para` 映射

| 参数名 | 含义 | 是否算法必填 | 说明 |
| --- | --- | --- | --- |
| `yw_space` | 液位控制带/目标间距 | 是 | 控制参数 |
| `yw_min` | 液位下限 | 是 | 控制参数 |
| `yw_max` | 液位上限 | 是 | 控制参数 |
| `kp_md_FL` | 分流阀 PID `kp` | 是 | 控制参数 |
| `ki_md_FL` | 分流阀 PID `ki` | 是 | 控制参数 |
| `kd_md_FL` | 分流阀 PID `kd` | 是 | 控制参数 |
| `kp_md_BS` | 补水 PID `kp` | 是 | 控制参数 |
| `ki_md_BS` | 补水 PID `ki` | 是 | 控制参数 |
| `kd_md_BS` | 补水 PID `kd` | 是 | 控制参数 |
| `update_ctrl` | 是否重置/更新控制器 | 是 | 首次启动通常为 `1` |
| `diverter_offset` | 分流阀偏置 | 是 | 控制修正量 |
| `water_offset` | 补水阀偏置 | 是 | 控制修正量 |
| `water_switch` | 补水反馈切换开关 | 是 | 决定读取 `data_short` 哪个位置 |

| `316` | `KMrhoSD_oldSys` | 旧系统设定密度 |

数据库连接写死在算法代码中：

- MySQL：`192.168.100.170:3306`
- 数据库：`ai_data`
- 表：`opcdata`

这意味着现场还要确认：

- Python 服务是否能访问该库
- 上述字段是否真实存在
- 是否继续沿用“算法侧查库”，还是改成由平台统一传入

### 4.6 智能密控设定密度预测接口

接口：`/predictRhoSD`

`data` 数组映射：

| 下标 | Python字段 | 含义 |
| --- | --- | --- |
| `data[0]` | `HF201` | 现场指标1，需确认点位名 |
| `data[1]` | `HF501` | 现场指标2，需确认点位名 |
| `data[2]` | `HF502` | 现场指标3，需确认点位名 |

- `pred_MMrho_SD`：中煤设定密度建议

这里代码里只有变量名，没有中文注释解释物理含义，现场必须再确认 `HF201/HF501/HF502` 对应的真实点位。

### 5.2 `data_long` 映射

| 下标 | Python字段 | 含义 | 数据类型 | 现场建议来源 |

| `data_long[0]` | `ZD` | 浊度历史序列 | 数值数组 | 浊度计 |

| 下标 | Python字段 | 含义 | 数据类型 |
| --- | --- | --- | --- |
| `data_short[0]` | `pump` | 当前加药泵频率 | 数值数组或短窗数组 |
| `data_short[1]` | `MN_in` | 煤泥入料 | 数值数组或短窗数组 |

### 5.4 `para` 映射

| 参数名 | 含义 | 是否算法必填 | 说明 |
| --- | --- | --- | --- |
| `ZD_SD` | 浊度设定值 | 是 | 控制目标 |
| `para1` | 控制参数1 | 是 | 具体物理意义需现场确认 |
| `para2` | 控制参数2 | 是 | 具体物理意义需现场确认 |
| `para3` | 控制参数3 | 是 | 具体物理意义需现场确认 |
| `update_ctrl` | 是否重置/更新控制器 | 是 | 首次启动通常为 `1` |
| `pumpNumber` | 主泵编号 | 是 | 算法返回时沿用该编号 |
| `JM_measured` | 界面仪实测值/校正值 | 是 | 现场需确认口径 |
| `backupPumpNumber` | 备用泵编号 | 否，但服务回写依赖 | Python `main.py` 写回备用泵时使用 |

说明：

- `backupPumpNumber` 不是算法控制器本体必填，但 Python 服务回写时会读取它。
- 前端演示模板中的 `density_setpoint`、`run_state` 不属于当前 Python 加药模型真实必填参数。

### 5.5 泵编号映射

| 编号 | 泵位 |
| --- | --- |
| `1` | 末煤1号给定频率 |
| `2` | 末煤2号给定频率 |

| `4` | 新加药2号给定频率 |
| `5` | 块煤1号给定频率 |
| `6` | 块煤2号给定频率 |

这套映射需要现场再确认一次，特别是 `5201/5202` 是否和块煤 1/2 号泵完全对应。

## 6. 现场必须确认的共性问题

### 6.1 时间序列口径

- 每个数组需要多少个点
- 采样周期是秒级、10 秒级还是分钟级
- 时间顺序是旧到新，还是新到旧
- 缺失值、坏点、断点怎么补

### 6.2 点位信息

- OPC Tag/PLC 地址
- 点位中文名称
- 单位
- 量程
- 是否模拟量/开关量
- 实时值来源系统
- 历史值来源系统

### 6.3 控制闭环归属

当前 Python 服务中已经存在直接写 Node-RED/OPC 的逻辑，因此现场必须拍板：

- 是继续由 Python 服务直接回写
- 还是由 Java 平台统一接管回写

如果不先定这个边界，现场联调会反复改接口。

## 8. 当前已识别风险

- 风险1：前端演示模板与算法真实字段不一致。
- 风险2：智能密控依赖算法侧本地 MySQL 查库，不是纯接口入参。
- 风险3：`HF201/HF501/HF502`、`para1/para2/para3`、`JM_measured` 的现场业务含义还不完整。
- 风险4：回写逻辑目前在 Python 服务内，平台与算法的职责边界还没最终定。

## 9. 最终建议

- 现场要数时，直接把本文件第 7 节发给现场人员填写。
- 联调前，先把第 6 节的共性问题一次性确认完。
- 如果后续决定由平台统一回写，我建议再做一版“点位映射配置表”，把 Python 里的写死点位彻底移出代码。
