# 管控平台部署指南

本文档同时覆盖两种部署方式：

1. `Docker Compose` 一键部署
2. `Windows 原生安装` 部署

现场建议：

- 优先使用 `Docker`，构建、升级和回滚都更省事。
- 如果现场因为安全策略、权限、内核版本、运维规范等原因无法使用 Docker，就按本文第 4 节开始走原生安装。

---

## 1. 部署方式选择

### 1.1 推荐方式

推荐使用项目根目录的：

- [docker-compose.yml](/d:/金海泽地/代码/管控平台/docker-compose.yml)
- [deploy/docker-up-all.ps1](/d:/金海泽地/代码/管控平台/deploy/docker-up-all.ps1)
- [deploy/docker-up-all.bat](/d:/金海泽地/代码/管控平台/deploy/docker-up-all.bat)

适用场景：

- 现场允许安装 Docker
- 希望快速整套上线
- 需要把 `PostgreSQL`、前端、后端、智能密控、智能加药、密控 MySQL 一起装进容器

### 1.2 无法使用 Docker 时

改走 `Windows 原生安装`：

- 后端：`Java 17 + Spring Boot JAR`
- 前端：`Nginx + web/dist`
- 主数据库：`PostgreSQL`
- 模型服务：`Python 3.10` 直接运行
- 可选：智能密控如果坚持走旧逻辑，再单独安装一个 `MySQL`

## 2. Docker 部署

### 2.1 前置条件

- 已安装 `Docker Desktop` 或 `Docker Engine + Compose`
- 主机端口可用：
  - `80`
  - `8080`
  - `5432`
  - `6788`
  - `6789`
  - `3306`

### 2.2 启动

项目根目录执行：

```powershell
.\deploy\docker-up-all.ps1
```

或者：

```powershell
docker compose --env-file .env.docker --profile models up -d --build
```

### 2.3 访问地址

| 服务 | 地址 |
| --- | --- |
| 前端 | `http://localhost` |
| 后端 | `http://localhost:8080` |
| PostgreSQL | `localhost:5432` |
| 智能加药 | `http://localhost:6788` |
| 智能密控 | `http://localhost:6789` |
| 密控 MySQL | `localhost:3306` |

### 2.4 常用命令

```powershell
docker compose ps
docker compose logs -f server
docker compose logs -f web
docker compose logs -f smart-density
docker compose logs -f smart-reagent
docker compose down
```

## 3. Docker 模型说明

### 3.1 智能密控

当前已经支持两种模式：

1. `param` 模式：优先从接口参数或默认值取设定密度
2. `mysql` 模式：从容器内 `smart-density-mysql` 读取旧系统设定密度

相关环境变量在：

- [.env.docker.example](/d:/金海泽地/代码/管控平台/.env.docker.example)

关键变量：

```env
SMART_DENSITY_RHO_SOURCE=param
SMART_DENSITY_DEFAULT_3207=1.43
SMART_DENSITY_DEFAULT_3208=1.47
SMART_DENSITY_DEFAULT_316=1.38
```

如果现场要求必须走旧 MySQL：

```env
SMART_DENSITY_RHO_SOURCE=mysql
SMART_DENSITY_DB_HOST=smart-density-mysql
SMART_DENSITY_DB_PORT=3306
SMART_DENSITY_DB_NAME=ai_data
SMART_DENSITY_DB_TABLE=opcdata
SMART_DENSITY_DB_USER=smart_density
SMART_DENSITY_DB_PASSWORD=smart_density_change_me
```

### 3.2 智能加药 / 智能密控回写

两个模型的回写地址现在都改成了环境变量：

```env
MODEL_WRITEBACK_URL=http://host.docker.internal:1880/Post_mikong
```

如果现场没有回写链路，模型仍可启动，但调用时可能打印回写异常日志。

## 4. 无法使用 Docker 时的安装部署

这一节就是现场“不能上 Docker”时的正式安装步骤。

## 4.1 需要安装的软件

### 4.1.1 Java 17

用途：

- 运行后端 `Spring Boot`

要求：

- 建议 `Temurin/OpenJDK 17`

安装完成后验证：

```powershell
java -version
```

### 4.1.2 Maven 3.9+

用途：

- 构建后端 JAR

如果现场机器不编译，只运行已打好的包，也可以不装 Maven。

验证：

```powershell
mvn -v
```

### 4.1.3 Node.js 22+

用途：

- 构建前端 `web/dist`

如果现场机器不编译，只部署已经打好的 `dist`，也可以不装 Node。

验证：

```powershell
node -v
npm -v
```

### 4.1.4 PostgreSQL 15/16

用途：

- 平台主数据库

建议创建数据库：

```text
库名: coal_platform
用户: coal_user
密码: 现场自定

```

### 4.1.5 Nginx for Windows

用途：

- 托管前端静态文件
- 反向代理后端 `/api`

部署时使用：

- [deploy/nginx.conf](/d:/金海泽地/代码/管控平台/deploy/nginx.conf)

### 4.1.6 Python 3.10

用途：

- 运行 `smart-density`
- 运行 `smart-reagent`

验证：

```powershell
python --version
pip --version
```

### 4.1.7 可选：MySQL 8.x

仅在以下情况下需要：

- 智能密控必须沿用旧逻辑
- 需要从 MySQL 读取 `3207rhoSD_oldSys`、`3208rhoSD_oldSys`、`KMrhoSD_oldSys`

如果现场不想再装 MySQL，就把密控改为 `param` 模式即可，不必安装。

## 4.3 原生安装顺序

建议按下面顺序做：

1. 安装 `PostgreSQL`
2. 初始化主库
3. 部署后端 JAR
4. 部署前端 `dist + nginx`
5. 启动后端
6. 启动 Nginx
7. 最后再启动智能密控、智能加药

不要一开始就先上模型服务，先把平台主链路跑通。

## 4.5 后端安装

### 4.5.1 构建后端

如果现场需要编译：

```powershell

mvn clean package -DskipTests
```

输出文件：

- [server/target/platform-server-0.0.1-SNAPSHOT.jar](/d:/金海泽地/代码/管控平台/server/target/platform-server-0.0.1-SNAPSHOT.jar)

### 4.5.2 复制后端文件

复制到：

```text
D:\platform\server\
```

需要复制：

- `server/target/platform-server-0.0.1-SNAPSHOT.jar`
- `server/src/main/resources/application-prod.yml`
- [deploy/coal-platform-service.xml](/d:/金海泽地/代码/管控平台/deploy/coal-platform-service.xml)

重点修改：

- `spring.datasource.url`
- `spring.datasource.username`
- `spring.datasource.password`

- `platform.iot.kepserver.password`

- `smart-density.base-url`
- `smart-reagent.enabled`
- `smart-reagent.base-url`


验证：

### 4.5.5 注册 Windows 服务

```powershell

```

常用命令：


## 6. 常见问题

### 6.1 后端启动失败

优先检查：

- Java 版本是否为 17
- PostgreSQL 连接串、用户名、密码是否正确
- `application-prod.yml` 是否指向正确环境

### 6.2 前端空白

优先检查：

- `web/dist` 是否完整
- Nginx `root` 路径是否指向 `D:/platform/web/dist`
- Nginx 是否已重载

### 6.3 模型服务起不来

优先检查：

- Python 是否为 3.10
- `pip install -r requirements.txt` 是否成功
- 端口 `6788`、`6789` 是否被占用

### 6.4 密控调用失败

优先检查：

- 传入参数是否完整
- 如果走 `mysql` 模式，MySQL 是否连通
- 如果不想依赖 MySQL，是否已改为 `param` 模式

### 6.5 Docker 不能用时的最小上线方案

如果现场时间紧，建议先这样：

1. PostgreSQL 安装好
2. 后端 JAR 跑起来
3. 前端静态页通过 Nginx 发布
4. 模型先不开或只开 `smart-reagent`
5. 智能密控后补

这样至少平台主功能可以先上线。
