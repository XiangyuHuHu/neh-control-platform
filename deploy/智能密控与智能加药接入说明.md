# 智能密控与智能加药接入说明

## 当前集成方式

当前管控平台没有把 `卓朗2` 中的 Python 模型直接嵌入 Spring Boot。

采用的是更稳的集成方式：

1. Python 模型服务独立运行
2. 当前 Java 后端提供统一代理接口
3. 前端只调用当前管控平台接口
4. Python 服务不可用时，后端自动回退到演示数据

这样做的原因：

- Python 模型依赖 `fastapi`、`tensorflow/keras` 等运行环境，不适合塞进 Java 进程
- 后续现场切换模型地址时，只改配置，不改页面结构
- 现场算法升级时，不需要重新打包整个管控平台

## 现场部署建议

### 推荐结构

1. `管控平台后端` 单独部署
2. `智能密控 Python 服务` 单独部署
3. `智能加药 Python 服务` 单独部署
4. 通过内网 HTTP 调用

### Windows 启动脚本

当前仓库已补充：

- [start-smart-density-model.bat](d:/金海泽地/代码/管控平台/deploy/start-smart-density-model.bat)
- [start-smart-reagent-model.bat](d:/金海泽地/代码/管控平台/deploy/start-smart-reagent-model.bat)

使用前只需要确认两件事：

1. 脚本中的 `MODEL_ROOT` 指向当前仓库内模型目录
2. 运行机器可直接执行 `python`

启动后可用以下接口检查模型服务状态：

- `GET /api/smart-density/health`
- `GET /api/smart-reagent/health`

### 不推荐结构

不建议：

- 把 Python 模型代码手工改写成 Java
- 把 Python 服务直接塞进 Spring Boot 同一个进程
- 把现场 OPC/Node-RED 写入逻辑直接前移到前端

原因是：

- 维护成本高
- 升级困难
- 现场排障会更复杂

## 后续需要补的数据工作

当前页面已经具备功能骨架，但要接现场真数据，还需要补：

1. 各单元点位与 `dataLong / dataShort / params` 的字段映射
2. 模型输入数组顺序定义
3. OPC/PLC 回写开关是否由 Python 服务负责，还是转到 Java 后端统一回写
4. 模型失败时的告警与人工接管策略

## 结论

当前最合理的融合方式已经落地：

- 页面入口已接入当前管控平台
- 后端接口已统一代理
- Python 模型保留独立服务形态
- 后续接现场时只需要补配置和字段映射，不需要重做页面结构
